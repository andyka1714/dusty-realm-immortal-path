# 即時戰鬥改造分析

本文件回答一件事：  
如果現在的戰鬥從「碰撞後播放戰報」改成「地圖上直接攻擊、帶射程的即時對戰」，應該怎麼改才合理。

---

## 1. 目前戰鬥的真實架構

目前流程曾經是：

1. 玩家在地圖上碰到怪
2. `adventureSlice` 進入 `isBattling`
3. `Adventure.tsx` 呼叫 `runAutoBattle()`
4. 一次算完整場戰鬥
5. 再把 `logs` 拿來回放

現在已經往前推進到：

- 地圖接戰已支援距離觸發
- `runAutoBattle()` 已改成時間軸模擬
- 每筆戰鬥事件會帶 `timeMs`
- 戰報回放會依事件間隔播放，不再固定每筆 0.5 秒

也就是說：

- 現在仍不是完整即時戰鬥
- 但已經不是純回合制資料結構
- 本質上是「先做時間軸模擬，再把事件依時間播給玩家看」

這種做法的優點：

1. 簡單
2. 容易做數值驗證
3. 容易做 log

缺點也很明顯：

1. 沒有臨場感
2. 技能 CD、攻速、位移都只是文字敘述
3. 很難自然接上更即時的技能施放感

### 1.1 已完成的第一階段

目前已完成的第一階段已經不只是「距離接戰」，而是：

- 地圖接戰已支援「距離觸發」，不再只有同格碰撞
- 玩家鎖定怪物後，進入職業接戰距離即可直接在場景內普攻或施法
- 怪物不再靠 proximity 自動切戰報，而是會在場景內依射程直接回擊
- 玩家血量、護盾、目標血量、狀態與技能冷卻，已在場景 HUD 直接顯示
- 地圖內即時擊殺已可直接結算修為、靈石與掉落，不再必須透過舊戰報流程收尾
- 戰鬥結果會綁定實際怪物實例，勝利後刪除正確那一隻怪
- 舊的 `battle modal / isBattling` 流程已退成相容尾端，不再是主要遊玩流程
- 玩家地圖內直接出手已開始吃 `executionTimeMs`，命中、浮字與掉落結算會延後到前搖結束後才發生
- 怪物已正式支援：
  - `resistances`
  - `weaknesses`
  - `affixes`（欄位已預留，正式詞綴行為仍待補）
- 玩家的五行輸出現在會吃到怪物的元素弱點 / 抗性乘區，不再只有屬性克制一層判定
- 第一批高境界被動已接入專屬行為，而不再只剩通用屬性加成：
  - 護體劍罡：致命抵擋 + 反震
  - 滴血重生：缺血越多回復越快
  - 法力源泉：持續回靈 + 高靈力增傷
  - 劍意化形：普攻雙段
  - 肉身成聖：重擊減半
  - 荒古戰體：定時震散負面狀態
  - 空間法則：部分傷害轉入虛空
  - 向死而生：低血量暴擊爆發
  - 滴血重生（真）：死亡逆轉
  - 仙法通神：術式回響
  - 不死不滅：保留最後 1 點生命

目前正式實作的玩家接戰距離：

- 劍修：`1`
- 體修：`1`
- 法修：`3`

目前已先示範接上遠程怪物的東線法術系怪物：

- `迷霧幻影`
- `湖畔水怪`
- `寒冰龜`
- `靈湖水蛟`
- `幻海魔鯨`

## 2. 你要的方向，其實不是純碰撞開戰

你這次補充後的方向比較接近：

- 地圖上直接攻擊
- 雙方站在場上對打
- 有攻擊距離
- 有攻速
- 有技能 CD
- 有技能施放節奏
- 遠程技能不一定要貼怪
- 有些怪物本身也會有較遠攻擊距離
- 初期仍可簡化操作，不一定要手操走位

這比較像：

- 即時自動戰鬥
- 半即時戰鬥

不是純回合制，也不是純手操動作遊戲。

我認為這個方向是合理的，而且很適合目前專案。

## 3. 我建議的戰鬥型態

### 3.1 第一階段：即時自動對戰

最適合你現在專案的，不是直接做全 ARPG，而是：

- 保留地圖上直接索敵與直接攻擊
- 保留角色與怪物自動攻擊
- 加入攻擊距離與索敵邏輯
- 加入攻擊間隔、技能 CD、施法節奏
- 在同一個戰鬥場景中即時播放

也就是：

- 角色站左側
- 怪物站右側
- 雙方不再輪流回合
- 而是各自依照攻速與 CD 行動

這樣的好處：

1. 保留目前數值系統的大部分資產
2. 只要把「回合」改成「時間軸」
3. 可以很自然把技能、攻速、施法動作、射程接進來

### 3.2 第二階段：再決定要不要做場內移動

如果第一階段穩了，再考慮：

- 近戰追擊
- 法修拉距離
- 技能投射物
- 範圍圈

我不建議現在直接跳到這一步，因為會把整個戰鬥系統一次打散。

## 4. 即時戰鬥和現在回合制最大的設計差異

### 4.1 現在是「每回合一次判定」

目前核心是：

- 玩家打一次
- 敵人打一次
- 再進下一回合

### 4.2 即時戰鬥會改成「時間軸驅動」

即時版應改成：

- 每個單位有自己的 `attackInterval`
- 每個單位有自己的 `attackRange`
- 每個單位有自己的 `acquireRange`
- 每個技能有自己的 `cooldownSeconds`
- 每個技能有自己的 `castRange`
- 每個效果有自己的 `durationSeconds`
- 遊戲每個 tick 更新一次

所以戰鬥引擎應該從：

- `runAutoBattle(player, enemy) -> 一次回傳整包結果`

改成：

- `createBattleState(player, enemy)`
- `tickBattle(state, deltaMs)`
- 每次 tick 更新血量、技能 CD、動畫事件

## 5. 射程與索敵應該怎麼設計

這一塊是你這次補充後，最應該加進來的新核心。

### 5.1 我建議每個單位至少要有三個距離參數

#### `attackRange`

代表：

- 普攻可命中的距離

#### `skillRange`

代表：

- 技能能施放的距離

#### `acquireRange`

代表：

- AI 會主動追擊與鎖定的距離

### 5.1.1 已完成的技能資料接線

目前主動技能資料已補上即時戰鬥所需的核心欄位：

- `castRange`
- `cooldownSeconds`
- `castTimeMs`
- `areaShape`
- `areaRadius`
- `maxTargets`

目前練氣、築基的三職業主動技能已先補齊顯式資料；其餘高境界技能則在匯出技能表時自動補齊：

- `cooldownSeconds`
- `castRange`
- `castTimeMs`
- `projectileSpeed`
- `areaShape`
- `areaRadius`
- `maxTargets`

這代表高境界技能雖然還沒逐招手調，但已不是缺欄位狀態，而是進入「之後逐招精修」階段。

### 5.1.2 已完成的狀態效果與專屬效果第一批接線

目前時間軸戰鬥內核已正式支援並接上：

- `stun`
- `burn`
- `poison`
- `bleed`
- `shield`
- 第一版戰鬥 HUD 已可顯示玩家與怪物當前狀態圖示，資料來源為戰鬥事件快照
- `armorBreak`
- `reflect`
- 第一版遠程投射物已透過 `visualEffects` 回灌到地圖舞台
- 第一版施法前搖 / 聚氣圈已透過 `visualEffects` 回灌到地圖舞台
- 第一版範圍技能地面提示已透過 `visualEffects` 回灌到地圖舞台
- 第一版命中受擊爆點 / 震盪圈已透過 `visualEffects` 回灌到地圖舞台
- 戰鬥勝利時，怪物已會先播放第一版潰散脈衝，再從地圖移除
- 戰鬥 HUD 已補上第一版主動技能冷卻條與剩餘秒數
- `areaShape / areaRadius / maxTargets` 已正式接進目前的單體時間軸戰鬥，用於計算範圍技能在單體場景下的有效承傷係數
- 地圖內世界戰鬥也已補上第一版多目標命中判定：`circle / line / cone` 會依玩家與主目標的相對位置波及附近怪物，正式脫離「只有欄位、只有特效」狀態
- `AdventureStage` 的投射物 / AOE / 命中 / 施法特效參數也已抽成可測 helper，讓即時地圖戰鬥表現層不再只能靠畫面手動檢查
- `battleSystem` 內也已抽出技能與怪物特招的 timeline profile helper，世界戰鬥與 `runAutoBattle()` 現在開始共用 `executionTime / projectile / AOE / cooldown` 基準
- 技能與怪物特招的 status 建立、時間正規化、player / enemy 分側，也已開始收斂到 battleSystem 共用 helper，不再讓 world strike 與 timeline combat 各寫一套
- 玩家主動術式施加給敵方的控制、破甲追擊、狀態推入 / 戰鬥日誌，以及術式免耗、冷卻縮短、靈潮回補、劍心重置與法修築基層數累進，都已開始收斂到 `resolvePlayerSkillStatusApplication(...) / appendAndLogCombatStatuses(...) / resolvePlayerActiveResourceFlow(...)` 這組共用 resolver。
- `劍意化形 / 虛空劍陣 / 撒豆成兵` 這批追擊與召喚後續傷害，也已開始抽成 `applyPlayerEchoAndSummonFollowupEffects(...)`，縮小高境界 active follow-up 在時間軸主循環裡的 inline 分支。
- 開場初始護體、待命訊息與帝境初始特招延後，已開始整併到 `initializeCombatEncounter(...)`，降低 `runAutoBattle()` 主循環的前置鋪排負擔。
- Boss 破綻觸發與對應戰鬥事件，已開始整併到 `rollBossBreakOpportunity(...)`，主循環不再重複散寫同一段爆發窗口判定。

- 敵方被控跳過回合的流程，也已開始整併到 `resolveEnemyIncapacitatedTurn(...)`，控制跳過與 `養劍術` 疊層不再在主循環散寫。

- 開場被動狀態、待命訊息與 enemy special 起始延後，也已開始透過 `resolveInitialPassiveStateBundle(...)` 聚合，`initializeCombatEncounter(...)` 不再繼續維護一長串被動布林參數。

- 回合開場維護也已開始整併到 `resolveTurnStartMaintenance(...)`，狀態 tick、被動回復與淨化流程不再直接散在主循環開頭。
- 戰鬥開場 seed 流程也已開始整併到 `seedCombatEncounter(...)`，初始護體與 enemy special 起始冷卻不再直接在主循環鋪開。

- 戰鬥基礎設施的 snapshot / tick wiring，也已開始整併到 `createCombatInfrastructure(...)`，snapshot provider 與 status tick processor 不再在 `runAutoBattle()` 內各自散寫。
- 戰鬥環境 seed 的回寫，也已開始整併到 `applyPreparedCombatLoopState(...)`，開場補入的護體狀態與 enemy special 延後不再在主函式逐欄覆寫。
- 初始 combat loop state 也已開始整併到 `createInitialCombatLoopState(...)`，hp/mp、冷卻、疊層、護體與回合旗標不再在 `runAutoBattle()` 開頭逐項手寫。
- 玩家出手階段也已開始整併到 `resolvePlayerActionPhase(...)`，週期護體、Boss 破綻與正式出手流程不再在主循環拆成兩段散寫。

### 5.2 對玩家職業的建議

#### 劍修

- `attackRange`: 1
- `skillRange`: 1 ~ 2
- `acquireRange`: 中等

#### 體修

- `attackRange`: 1
- `skillRange`: 1
- `acquireRange`: 中等偏短

#### 法修

- `attackRange`: 3 ~ 5
- `skillRange`: 4 ~ 7
- `acquireRange`: 最長

### 5.3 對怪物的建議

怪物也不該全部都只有近戰。

建議至少分成：

1. 近戰怪：`attackRange = 1`
2. 投射怪：`attackRange = 3 ~ 5`
3. 法術怪：`skillRange = 4 ~ 6`
4. Boss：可混合近遠兩套攻擊模式

這樣後面不同地圖、不同路線才會真的有體感差異。

## 6. 三職業在即時戰鬥中的差異建議

### 劍修

定位：

- 攻速最快
- 技能 CD 偏短
- 單體爆發高
- 有短距離突進或瞬斬能力

建議：

- 普攻速度最快
- 技能偏短 CD、多段斬擊、突進
- 對 Boss 有壓力輸出感

### 體修

定位：

- 攻速中等偏慢
- 技能 CD 中等
- 肉、穩、控制
- 偏近身壓制

建議：

- 普攻慢，但每一下較重
- 技能帶擊退、暈眩、減傷、護盾
- 不應該靠純坦就無腦磨死所有 Boss

### 法修

定位：

- 普攻頻率中等
- 技能才是主體
- 吃施法節奏與 CD 管理
- 射程最長

建議：

- 技能多為遠程、範圍、DOT、控場
- MP / 回靈 / CD 會成為主要體感
- 不應該只靠普攻輸出

## 7. 你這個專案最適合的第一版即時戰鬥

我建議第一版只做以下要素：

### 必做

1. 攻擊速度
2. 攻擊距離 / 技能距離
2. 技能 CD
3. 技能施放動畫事件
4. 即時更新血條
5. DOT / stun / freeze 這種持續效果
6. 索敵與追擊邏輯

### 先不要做

1. 複雜走位
2. 彈道碰撞
3. 場地 AOE 預警圈
4. 玩家手動施法輸入

原因很簡單：

- 你現在最需要的是把戰鬥從「假即時」變成「真即時」
- 不是一次直接做成大型 ARPG

## 8. 目前架構該怎麼改

### 7.1 現在要拆掉的核心假設

目前 `runAutoBattle()` 假設：

- 一次就算完整場戰鬥

未來要改成：

- `battleState`
- `timeline`
- `entities`
- `cooldowns`
- `statusEffects`
- `ranges`
- `targeting`

### 7.2 建議新增的模組

我建議新增：

- `types/realtimeBattle.ts`
- `utils/realtimeBattle.ts`
- `components/adventure/BattleArena.tsx`

其中：

- `realtimeBattle.ts` 負責戰鬥時間軸
- `realtimeBattle.ts` 負責時間軸、索敵、射程與 CD
- `BattleArena.tsx` 負責即時畫面呈現

### 7.3 轉換策略

不要直接刪 `runAutoBattle()`。

正確做法是：

1. 保留 `runAutoBattle()` 當平衡驗證工具
2. 新增 `realtimeBattle` 引擎
3. UI 切到新戰鬥場景

這樣你還是能用自動化測試去驗證數值，不會因為改成即時畫面就失去可測試性。

---

## 9. 我對第一版即時戰鬥的具體主張

如果要兼顧你要的沉浸感和目前專案可落地性，我建議第一版直接定成：

1. 地圖上直接攻擊，不靠碰撞才開戰
2. 角色與怪物都有攻速
3. 角色與怪物都有射程
4. 技能都有 CD 與施法距離
5. 先做自動索敵與自動戰鬥
6. 先不要做高操作量的手動技

這樣的版本已經會比現在的戰報回放好很多，而且仍然能保留數值可測試性。

## 10. 結論

你現在要的不是把回合制畫面做得更花，而是把戰鬥核心從「一次結算」改成「時間軸驅動」。

我建議的正確順序是：

1. 先把技能系統改成技能書取得
2. 同時保留 `runAutoBattle()` 當數值驗證工具
3. 再新增一套 `realtimeBattle`，把攻速、射程、CD、索敵一起做進去

這樣做最穩，也最不會把現有專案整個打爛。

# 3D 渲染與戰鬥呈現評估

本文件回答三件事：

1. 目前這個專案能不能改成 3D  
2. 用 `Three.js / WebGL` 是否合適  
3. 如果後續要做真正的即時戰鬥，畫面層應該怎麼演進

---

## 1. 結論先講

### 1.1 技術上做得到

可以。

目前專案的地圖、怪物、玩家位置，本質上都已經是格子座標：

- 玩家：`Coordinate`
- 怪物：`ActiveMonster`
- 地圖：`MapData`

這種資料本來就可以映射成：

- 2D Canvas / Pixi
- 2.5D 偽 3D
- 真 3D 場景

所以「能不能做」不是問題。

### 1.2 但現在不適合直接全面切 3D

我不建議現在直接把主場景全面改成 `Three.js` 真 3D，原因很直接：

1. 目前玩法主體還在快速變動  
2. 戰鬥才剛從回合式資料結構改成時間軸  
3. UI / 面板 / 技能 / 掉落 / AI 還在持續重構  
4. 美術資產目前不是 3D pipeline

如果現在直接跳 3D，最容易發生的是：

- 視覺層先變複雜
- 遊戲核心還沒穩
- 開發速度明顯下降
- 後面每改一個系統都要連動場景表現

所以從產品節奏來看，現在直接全面 3D 不划算。

---

## 2. 目前專案的最佳演進方向

### 2.1 先走 2D / 2.5D 強化版

現在最合理的方向不是立刻切 3D，而是：

- 保留目前 `PixiJS` 主場景
- 把目前格子移動、怪物、技能表現做得更像 ARPG
- 用 2.5D 視覺語言提升沉浸感

也就是：

1. 地圖仍是平面
2. 角色和怪物仍是 sprite / icon / 特效層
3. 加入：
   - 攻擊動畫
   - 投射物
   - 範圍圈
   - 命中特效
   - 血條
   - 技能冷卻與施法提示

這樣做的好處：

1. 和現在資料結構最相容
2. 可快速驗證玩法
3. 後續若要升級成 3D，也不會白做

### 2.2 等戰鬥核心穩定後，再評估局部 3D

比較合理的順序是：

1. 先把即時戰鬥完整做穩
2. 再把地圖視覺升級成 2.5D
3. 最後只把「主場景」評估改成 3D

這樣風險最低。

---

## 3. `Three.js / WebGL` 合不合適

### 3.1 合適的部分

如果你未來要做：

- 可旋轉場景
- 立體角色
- 3D 地圖
- 粒子法術
- 浮空島 / 洞府 / 宗門全景

那 `Three.js` 是合理的。

它可以做：

- 3D 地形
- 角色模型
- 粒子技能
- 相機運鏡
- 即時光影

### 3.2 不合適的部分

對現在這個專案來說，`Three.js` 的主要問題不是功能，而是成本：

1. 場景管理會複雜很多
2. UI 和 3D 場景要重新分層
3. 目前沒有 3D 角色與怪物資產
4. 調參與除錯成本會明顯提高
5. 手機與低階裝置效能會更敏感

所以現在切 3D，會讓你把大量時間花在：

- 場景技術
- 相機
- 材質
- 模型流程

而不是花在「把遊戲變好玩」。

---

## 4. 現在的戰鬥模式如果往即時走，應該怎麼調整

### 4.1 現在已完成的基礎

目前已完成：

- 玩家有接戰距離
- 怪物有接戰距離
- 技能資料有 `castRange`
- 技能資料有 `cooldownSeconds`
- 戰鬥內核改成時間軸模擬
- 戰報回放會依事件時間播放

這代表底層已經不再是純回合制。

### 4.2 下一步應該做的不是 3D，而是這些

#### 第一層：戰鬥表現層

在 `PixiJS` 場景裡加入：

1. 玩家血條
2. 怪物血條
3. 普攻命中特效
4. 投射物
5. 範圍圈
6. 技能施放閃光 / 軌跡

#### 第二層：場上行為

把目前「接戰後切 modal」改成：

1. 同場景進入戰鬥狀態
2. 玩家與怪物維持一定距離
3. 依 `attackRange / castRange / cooldownSeconds` 執行
4. 技能直接在地圖場景裡表現

#### 第三層：更細的 AI 與射程

後續再補：

1. 近戰追擊
2. 遠程風箏
3. Boss 技能圈
4. 怪物分工
5. 投射物飛行時間

## 5. 如果未來真的要做 3D，我建議的方式

### 5.1 不要一次整包重寫

不要：

- 直接把 `AdventureStage` 全改成 3D

建議：

1. 保留 Redux / 資料層
2. 保留地圖與戰鬥資料模型
3. 新增一個獨立的 3D stage 原型
4. 只拿單一地圖驗證

也就是：

- `AdventureStage2D`
- `AdventureStage3DPrototype`

先雙軌，不要一刀切。

### 5.2 真正適合切 3D 的時機

我認為要到以下條件成立，才適合切 3D：

1. 即時戰鬥規則已穩
2. 三職業技能池已收斂
3. 怪物 AI 已穩
4. 地圖流程不再頻繁重改
5. 願意投入 3D 美術與效能優化成本

## 6. 我現在的建議

### 短期建議

現在先不要全面改 3D。  
先把現有 `PixiJS` 場景做成：

- 更快的移動
- 更好的戰鬥節奏
- 同場即時戰鬥
- 技能動畫 / 投射物 / 範圍圈

### 中期建議

當上面穩了，再做一個：

- 單地圖 `Three.js` 原型

只驗證：

- 視角
- 效能
- 操作手感
- UI 疊加方式

### 長期建議

如果 3D 原型證明真的值得，再逐步替換主場景，而不是現在就全面翻掉。

## 8. 單地圖 3D Prototype 技術方案

這一段不是全面切 3D，而是把 prototype 要怎麼做講清楚。

### 8.1 Prototype 範圍

建議只做一張地圖：

- `試劍台` 或 `三界戰場`

原因：

1. 兩者都有怪物、NPC、戰鬥與明確地圖邏輯
2. 不需要先碰整條世界地圖串接
3. 容易用現有 2D 資料直接映射成 3D 位置

### 8.2 技術切法

建議保留現有 Redux / 資料層，只新增一個獨立 prototype stage：

- `AdventureStage3DPrototype`

資料直接沿用：

- `MapData`
- `ActiveMonster`
- `Coordinate`
- `VisualEffect`

先驗證四件事：

1. 3D 相機視角是否比 2D 更好讀
2. 玩家移動與鎖定目標手感是否更好
3. 技能投射物 / AOE 在 3D 場景下是否更清楚
4. UI 疊在 3D 場景上是否仍可維持目前資訊密度

### 8.3 第一版渲染內容

只做最低限度：

1. 地板與地形：平面 mesh + height / normal 假效果
2. 玩家：簡單 capsule / placeholder sprite billboard
3. 怪物：簡單 capsule / icon billboard
4. 投射物：粒子或發光球
5. AOE：地面 decal / shader circle
6. 血條與狀態：仍走 React HUD 疊在畫面上

不要在 prototype 第一版就做：

- 完整角色模型
- 骨架動畫
- 複雜光影
- 大量場景物件破壞

### 8.4 驗證門檻

Prototype 要驗這些：

1. 桌機中階顯卡維持 `60 FPS`
2. 怪物數量 `30` 以上時仍能穩定
3. 投射物 / AOE / 狀態提示不會遮蔽操作
4. React HUD 與 3D 場景不會互搶控制

只要有任一項不過，就不應擴大 3D 範圍。

---

## 9. 若未來切 3D，需要先固定的資產與效能規格

### 9.1 資產規格

- 角色 / 怪物模型：低面數 stylized，優先單材質
- 動畫：待機 / 移動 / 普攻 / 施法 / 受擊 / 死亡 六大類
- 特效：投射物、命中、地面圈、護盾、DOT、控制狀態
- 地圖：可重複使用的 tile / cliff / altar / ruin 套件

### 9.2 動畫來源

可接受來源：

1. 自製簡化 placeholder
2. 商店授權低面數套件
3. Mixamo / 通用 humanoid 動畫作為早期驗證

正式量產前，要先決定：

- humanoid 骨架統一規格
- 非 humanoid 怪物如何共用動畫狀態機

### 9.3 效能預算

Prototype 階段建議上限：

- 同屏怪物：`30`
- Draw calls：盡量壓在 `150` 以下
- 特效同屏數：`20` 以下
- 單張貼圖尺寸：以 `512 / 1024` 為主

### 9.4 遷移順序

如果未來真的要切：

1. 先做單地圖 prototype
2. 再做單一路線高境界圖
3. 再做玩家 / 怪物戰鬥表現層
4. 最後才考慮全面取代 2D 主場景

---

## 7. 總結

目前專案：

- **技術上可以做 3D**
- **產品節奏上不適合現在就全面切 3D**
- **最合理的是先把 2D / 2.5D 即時戰鬥做完整**

也就是說，接下來最有價值的工作順序應該是：

1. 完成同場即時戰鬥
2. 補技能表現與投射物
3. 補怪物遠近戰 AI
4. 再做 3D 原型驗證

# 下一階段主線整合與優先級建議

本文件用來把目前已明確浮現的下一階段主線，從分散在多份審計文件、設計草稿與對話裡的方向，整理成同一份可執行的優先級建議。

這份文件承接：

- `14_整體改造Checklist.md` 的本輪大改造結案狀態
- `16_下一輪執行優先級Checklist.md` 已完成的 `P0 / P1 / P2`
- `輪迴轉生`、`洞府百業`、`事件 / 奇遇`、`宗門內容` 與 `像素風` 的下一輪方向盤點

---

## 1. 目前專案的真實狀態

前一輪主線目前已把下列基礎正式做穩：

1. battle core 已從舊回合式資料結構推進到時間軸與地圖內即時出手
2. 三職業核心技能池、技能書來源與高境界戰鬥欄位已正式收斂
3. 高境界怪物密度、路線主題掉落與 `仙帝` 端終盤門檻已做完一輪收口
4. `AdventureStage` 的 live combat presentation、怪物分工、狀態 cue 與危險區 visual language 已進入正式主流程

但如果從「產品下一步要補什麼，才能讓整個遊戲變成一個可長玩、可持續擴張的正式 loop」來看，缺口已經轉移到 battle core 以外：

1. `輪迴轉生` 有設計稿，但還不是正式系統
2. `洞府百業` 只有聚靈陣落地，煉丹與煉器仍是 placeholder
3. `事件 / 奇遇` 仍以年歲 flavor log 為主，尚未成為有選項與結果分支的正式 loop
4. `宗門與世界內容` 仍偏前期薄片，沒有中後期成長線與更多 route-specific 劇情承接
5. OpenSpec base spec 與 persistence 基線仍落後於目前真實玩法
6. 視覺方向若決定轉向 `像素風`，需要先有 pre-production 與資產規格，不應直接全量翻修

## 2. 六條候選主線

### A. 輪迴轉生正式化

目標：

- 把 `Soul Save / Merit / Lifetime Stats / Heirloom / Perks / Reincarnation Hall` 做成正式 meta progression
- 讓死亡從單純 reset，變成新一輪 build 的入口

目前狀態：

- `docs/02_Gameplay/reincarnation.md` 已經有概念與規格草稿
- `store/slices/characterSlice.ts` 目前的 `reincarnate` 只會把角色退回未初始化狀態，還沒有全域靈魂存檔、功德結算、遺產繼承與轉生配置

價值：

- 這是目前最缺的長線留存 loop
- 會直接決定死亡、壽元、後期 build 與多周目是否有意義

### B. 支撐型養成循環落地

目標：

- 把煉丹、煉器、洞府加成做成正式可迭代循環
- 讓後期乘區不只存在於 audit 表，而是真正進入可玩 loop

目前狀態：

- `pages/Workshop.tsx` 目前只有聚靈陣可用
- 煉丹爐與煉器台仍明確顯示尚未開放
- `01_修為與境界曲線審計.md` 已把高境界的丹藥 / 洞府 / 奇遇 / 事件倍率寫成正式規格

價值：

- 這是把高境界時長、裝備追求與資源消耗變成正式 progression 的關鍵
- 若沒有這條線，後期就會持續落回「只靠修煉 / 掛機 / 刷怪」的窄 loop

### C. 事件與奇遇系統

目標：

- 把目前年歲 flavor log 擴成正式 encounter system
- 讓事件具有選項、收益、風險、職業 / 路線差異與掉落 / 修為介面

目前狀態：

- `store/actions/timeActions.ts` 目前主要處理壽元警告、年歲 flavor log 與低機率頓悟
- 還沒有正式的事件池、選項、權重、結果與 route-specific rewards

價值：

- 這是把修仙遊戲做出「途中會發生什麼」的關鍵
- 也會直接承接後期奇遇倍率與內容辨識度

### D. 宗門與世界內容擴張

目標：

- 把現在的宗門與任務內容從前期入門，擴成可跨中後期的成長線
- 補地圖奇遇、專屬材料、里程碑 NPC 與境界段劇情

目前狀態：

- `data/quests.ts` 目前主要是 tutorial、三宗門入門與各自第一個試煉任務
- 中後期宗門線、境界里程碑事件與長鏈內容仍明顯不足

價值：

- 這條線會直接影響世界厚度與玩家的職業路線認同感
- 但它不是第一個該補的基礎，因為前面幾條 loop 還沒閉環

### E. Spec / Persistence / Migration 收口

目標：

- 把 OpenSpec base specs、persisted state migration 與正式文件回寫到目前真實玩法
- 讓戰鬥、世界、技能書與之後的新系統都有可信的基線

目前狀態：

- `openspec/specs/game-mechanics/spec.md` 的 `Purpose` 仍是 `TBD`
- spec 內對冒險與戰鬥的描述仍停在「網格探索與回合制戰鬥」
- persistence / migration 雖已有部分機制，但還沒有承接下一輪 meta progression 與跨周目資料

價值：

- 這條線不華麗，但它是之後所有正式 change 的地基
- 若不先補，後續輪迴、事件、工作坊與視覺重製都會建立在過時基線上

### F. 像素風主線

目標：

- 把目前遊戲的場景、角色、怪物、特效與 UI 語言統一到正式 `pixel-art` 方向
- 讓視覺、跨平台效能與內容產線轉向更穩定的 2D 資產模式

目前狀態：

- 專案前端目前建立在 `PixiJS`
- `AdventureStage` 與地圖資料本質上是格子 / sprite / overlay 的 2D pipeline
- 技術上適合像素風，但目前沒有正式的像素風資產規格、tileset bible、sprite 規則與 production pipeline

價值：

- 這條線是目前最現實、最適合 web / mobile 的視覺重製方向
- 但它不應被視為單一步驟的「整包換皮」，而應分成：
  1. `pixel-art pre-production`
  2. `pixel-art full rollout`

## 3. 重新評估：哪些主線真的該先做

### 3.1 評估標準

本文件用四個標準排優先級：

1. 是否直接補上產品的長線閉環
2. 是否會影響後續多條主線的正確性
3. 是否能在不大幅重做既有 battle core 的前提下持續擴張
4. 是否應該先在玩法穩定後再投資大量美術 / 內容成本

### 3.2 核心判斷

#### 第一判斷：最缺的是 loop，不是更多戰鬥 polish

目前最核心的問題，已不再是 battle core 不夠完整，而是：

- 死亡沒有第二層意義
- 後期乘區沒有對應玩法來源
- 時間流逝與世界遭遇還不是正式內容機制

這代表下一輪的優先級，應先放在 `輪迴 / 百業 / 事件`，不是直接繼續擴戰鬥。

#### 第二判斷：Spec / Persistence 是所有新系統的地基

如果不先回收 OpenSpec base spec 與 persistence / migration 基線：

- 輪迴資料會沒有正式永久層規格
- 百業與事件很容易做成只存在 UI 或 state 的局部功能
- 後續 change proposal 會繼續建立在已過時的 base spec 上

因此這條線雖然不一定最先帶來玩家體感，但優先級不能低。

#### 第三判斷：像素風值得做，但不能在 loop 沒定型前全量開工

像素風比真 3D 適合現在的 repo，也更符合 web / mobile。

但像素風的真正成本不在 renderer，而在：

1. 資產規格
2. tileset / sprite / VFX bible
3. UI 語言重定
4. 後續所有內容擴張都要跟這套風格對齊

因此像素風主線應該高於「單純視為遠景構想」，但低於「先把產品長線 loop 補起來」。

## 4. 建議優先級總表

### Priority 1. Spec / Persistence / Migration 收口

這是第一順位，不是因為它最有體感，而是因為它是之後所有正式系統的共用地基。

要收的內容包括：

1. 更新 `openspec/specs/game-mechanics/spec.md`，把即時戰鬥、技能書、路線掉落、世界戰鬥真相回寫
2. 盤點是否需要新增 `client-persistence` / `client-interface` 的對應規格
3. 盤清 persisted state 的版本遷移策略，讓之後 `Soul Save / Heirloom / Perks / Workshop / Events` 都有正式存檔承接

理由：

- 沒這一步，後面四條玩法主線都會疊在過時規格上
- 這條線的風險最低、收益是全面性的

### Priority 2. 輪迴轉生正式化

這是第二順位，因為它是整個遊戲最重要的 meta progression 缺口。

要收的內容包括：

1. `Soul Save`
2. `Merit` 結算
3. `Lifetime Stats`
4. `Heirloom`
5. `Perks`
6. `Reincarnation Hall`

理由：

- 這條線一落地，死亡、壽元、成長曲線、後期失敗成本才會正式成立
- 它會直接改變整個產品的長玩價值

### Priority 3. 支撐型養成循環落地

這是第三順位，因為輪迴之後，下一個最需要補的是「後期日常到底在做什麼」。

要收的內容包括：

1. 煉丹正式化
2. 煉器正式化
3. 洞府加成體系深化
4. 對應材料與產出循環
5. 與高境界倍率表對齊

理由：

- 後期乘區目前有規格，但沒有對應 loop
- 如果跳過這條線，後面奇遇與內容擴張仍會缺少穩定的資源消耗與 build 追求

### Priority 4. 事件與奇遇系統

這是第四順位，因為前面先補了 meta progression 與支撐型成長後，才值得把「途中遭遇」做成正式內容層。

要收的內容包括：

1. 正式 encounter pool
2. 選項與結果
3. 風險 / 收益
4. 職業 / 路線差異
5. 與修為、掉落、材料、狀態掛鉤

理由：

- 這條線會顯著提升修仙感與世界感
- 但若沒有前面的輪迴與百業承接，事件只會變成額外 flavor，而不是真正的 loop

### Priority 5. 像素風主線（先做 pre-production，不直接 full rollout）

這是第五順位，但要注意這裡的意思不是「把像素風整包往後丟」，而是：

- `pixel-art pre-production` 應在這個順位開始
- `pixel-art full rollout` 不應現在就整包展開

`pre-production` 應包含：

1. 像素風 art bible
2. tile / sprite / UI / VFX 尺寸規格
3. 一張地圖 + 一個玩家 + 兩隻怪 + 一套戰鬥 cue 的 vertical slice
4. web / mobile 的效能驗證
5. 自主生成素材的可行上限驗證

理由：

- 像素風很適合現在的技術架構
- 但如果在玩法長線還沒閉環前直接全面美術翻修，之後很容易邊做系統邊返工資產
- 先做 prototype，才能判斷要不要全面切換

### Priority 6. 宗門與世界內容擴張

這條線排第六，不是因為不重要，而是因為它最依賴前面幾條線先站穩。

理由：

- 宗門任務、中後期內容與地圖奇遇一旦擴張，就會連帶放大：
  - meta progression
  - workshop 材料需求
  - event / encounter rewards
  - 視覺風格的資產成本
- 如果前面幾條線還沒定型，現在大規模擴內容，之後返工機率最高

因此比較合理的順序是：

1. 先補系統地基與閉環
2. 再把內容量往外擴

## 5. 建議執行方式：不要把六條線同時全開

### 5.1 建議拆成三個階段

#### Phase 1. Foundation

包含：

1. `Spec / Persistence / Migration`
2. `輪迴轉生正式化`

目標：

- 建立下一輪所有系統的正式資料基線與永久成長層

#### Phase 2. Loop Closure

包含：

1. `支撐型養成循環落地`
2. `事件與奇遇系統`

目標：

- 讓後期乘區、年歲流逝、材料與修為增益真的形成日常 loop

#### Phase 3. Content and Visual Direction

包含：

1. `像素風 pre-production`
2. `宗門與世界內容擴張`

目標：

- 先確立正式視覺方向，再把中後期內容系統性擴張

### 5.2 像素風的特別處理

像素風不應和其他五條主線被視為同一種性質的工作。

其他五條主線主要是在補：

- 系統閉環
- 正式資料層
- 長線內容循環

像素風則是在決定：

- 長期視覺資產方向
- UI / 場景 / 怪物 / 玩家 / VFX 的整體風格

因此像素風更像一條高影響力的 production track，不該在沒有 prototype 的情況下直接變成全專案主路徑。

---

## 6. 最後結論

如果要把目前所有下一步主線放在同一張 priority table，建議順序如下：

1. `Spec / Persistence / Migration 收口`
2. `輪迴轉生正式化`
3. `支撐型養成循環落地`
4. `事件與奇遇系統`
5. `像素風主線（先做 pre-production / vertical slice）`
6. `宗門與世界內容擴張`

這個排序背後的原則不是「哪個最酷就先做」，而是：

1. 先補地基
2. 再補長線循環
3. 再決定正式視覺方向
4. 最後擴內容量

若要直接落地成下一輪 OpenSpec change，我建議第一條大主線不是直接開像素風 full conversion，也不是先大擴 quest，而是：

- `add-meta-progression-foundation`

它至少要先涵蓋：

1. `Spec / Persistence / Migration`
2. `輪迴轉生正式化`

等這條地基主線站穩後，再依序開：

- `add-workshop-and-event-loops`
- `evaluate-pixel-art-vertical-slice`
- `expand-sect-and-world-content`

這樣整體返工風險最低，也最符合目前 repo 的真實狀態。

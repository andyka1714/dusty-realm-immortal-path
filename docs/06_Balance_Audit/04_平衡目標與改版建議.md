## 2. 目前可以視為已站穩的部分

- 修為節奏、後期乘區、追趕機制與 TTK 目標，已有正式 registry 與測試
- 戰鬥公式、裝備總值區間、Boss 門檻與高境界內容密度，已有正式測試帶
- 技能池已收斂成 `formal core / battle-absorbed / retirement-ready` 視角
- `runAutoBattle()` 已拆出 runtime、loop、outcome、tick、開場與 action phase helper，不再是單一長函式
- `GameTooltip / GameHintBubble / GamePanel / Modal / GameSection` 已成為主要 UI 殼層語言

換句話說，現在不需要再回頭重做底層支架，接下來重點是把尾端主線正式結案。

## 3. 本輪已結案的主線

### 3.1 戰鬥

- 地圖世界戰鬥、舊戰報 replay 與 `runAutoBattle()` 已回收到同一套 battle core
- world strike / timeline combat / replay 的 queue、controller、preview、outcome、reward、cleanup 與 lifecycle 已集中到 battle core 專責模組，而 `battleSystem.ts` 已退成 barrel / shared type 入口
- checklist 最後一條 battle 主線已可視為正式結案；後續若再拆 battle core，只屬於下一輪優化，不再是這輪遺留

## 4. 下一輪若要續修的順序

### 第一優先：battle core 模組化優化批

1. 沿 `battleLog / battlePassives / battleStatuses / battleStatusEffects / battleProfiles / battleTiming / battleEncounter / battleTimeline / battleRewards / battleReplay / battleLifecycle / battleWorldStrike / battleWorldController / battleTurnPhases / battleTimelineResults` 的模組邊界繼續拆細 battle core
2. 只處理 battle core 內仍偏大的 orchestration，避免再把純函式塞回 `Adventure` 頁面外殼
3. 若 battle 規則有新增差異，優先補測試與正式文件，而不是再把「已共用 / 未共用」改回雙軌敘事
4. 下一輪追蹤清單已另開 [15_battle_core下一輪收尾Checklist](./15_battle_core下一輪收尾Checklist.md)，不再回寫到本輪 checklist

### 第二優先：文件與驗證批

1. 維持 `README / Gameplay / World / Audit` 對 battle 單一引擎狀態的單一說法
2. 用測試持續約束 world / timeline / replay 後續新增行為的差異
3. 若有新一輪 battle 擴充，再另外開 checklist 條目，不回頭把本輪結案項目改回未完成

---

## 5. 這份文件的結論

現在最不需要做的，是再回到「重拉一輪整體數值」。

現在真正該做的是：

1. 以目前已合流的 battle core 為基礎，繼續做模組化與可維護性整理
2. 讓新功能沿 battle core 擴張，而不是重新分叉 world / timeline / replay 三套規則
3. 把這輪 checklist 視為正式封存完成，後續需求另外開新主線追蹤

這輪平衡改造目前已可視為正式完成；之後若再動 battle，屬於下一輪優化。

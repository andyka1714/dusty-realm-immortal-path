# 五線內容深度執行計畫

本文件承接 `17_下一階段主線整合與優先級建議.md`，把下一階段要執行的五條線整理成可逐階段提交、逐階段驗證的工作批次。

目前前置狀態：

- `openspec list` 顯示既有 21 條 change 全部已完成
- `Encounter / Workshop / Sect World / Pixel Terrain / Base Spec` 的第一輪基線都已進主流程
- 玩家、怪物、NPC 仍維持文字 token；像素化只推進 terrain / background
- 每一階段完成後都要單獨驗證與提交

---

## 1. 執行總順序

### Phase 1: 事件內容庫擴張

Change id: `expand-encounter-content-library`

優先原因：

- 現在 encounter 機制已完整，但內容厚度不平均
- `元嬰 -> 合體 -> 大乘` 的可重複 route-specific 事件偏少
- 這條線可以先補內容與 regression，不需要先改 persisted state shape

交付目標：

- 建立 encounter coverage matrix
- 補中後期通用 / 職業 / 宗門 / 里程碑事件
- 新增內容覆蓋 regression
- 同步 encounter 文件

### Phase 2: 宗門與世界章節分支

Change id: `expand-sect-world-story-branches`

優先原因：

- 三宗 `task_04` 已把玩家推到化神前哨，但 `120 / 121 / 130+` 還缺 map-local NPC 與章節任務
- 這條線能把事件內容庫、宗門身份與世界地圖串成更完整的後段故事

交付目標：

- 補 `task_04` 之後的世界章節任務
- 補三界戰場 / 隕仙深淵等地圖 NPC 或對話掛點
- 新增 world-story regression
- 同步 `sect-world` 與 world docs

### Phase 3: Workshop 專精樹與材料 sink

Change id: `update-workshop-specialization-tree`

優先原因：

- Workshop 目前已有高階 recipe 與扁平專精，但還不是專精樹
- 這條線會碰 `WorkshopState` 與 migration，應該放在內容型 change 之後獨立做

交付目標：

- 建立煉丹 / 煉器專精樹資料模型
- 補前置節點、互斥分支、reset UI 與成本
- 擴充高階 route-specific material sink
- 補 persisted state migration 與 regression

### Phase 4: Pixel terrain landmark polish

Change id: `update-pixel-terrain-landmark-polish`

優先原因：

- terrain metadata 已有 `semanticRole / skeletonId`，但正式繪製仍偏通用方塊
- 這條線只處理背景地圖，不碰玩家、怪物、NPC、HUD 或 combat overlay

交付目標：

- 讓 `AdventureStage` 依 semantic role 畫出更清楚的 landmark / corridor / arena
- 補代表地圖 table-driven terrain regression
- 視需要補桌機 / 手機視覺回歸
- 更新 `pixel_art_bible`

### Phase 5: OpenSpec archive 與基線清理

Change id: 不另開功能 change，使用 OpenSpec archive 流程

優先原因：

- 目前 completed changes 仍大量留在 `openspec/changes/`
- 應等 Phase 1-4 實作完並回寫 specs 後，再統一封存，避免把 active work 混進 archive

交付目標：

- 檢查 `client-interface / client-persistence / game-mechanics` 是否吸收完成狀態
- 封存 completed changes
- 精簡 `16 / 17` 類 tracking 文件，把未來 backlog 移到新的入口
- 驗證 `openspec validate --strict`

## 2. Sub-agent 分工

本批次採用「主線總控 + 專線 sub-agent」模式：

- Encounter agent: 盤點 encounter 覆蓋率、事件內容矩陣、事件 regression
- Sect/world agent: 盤點 `task_04` 後章節、map-local NPC、世界里程碑
- Workshop agent: 盤點專精樹、材料 sink、migration 與 UI
- Pixel terrain agent: 盤點地圖背景 polish、actor-token guard、視覺回歸
- OpenSpec cleanup agent: 盤點 completed changes、base spec 吸收與 archive 指令

實作階段每條 change 會再建立對應 worker；worker 必須只負責該 change 的檔案範圍，不得回退其他 change 的修改。

---

## 3. 驗證與提交規則

每個 Phase 完成時都必須至少完成：

- 對應 `openspec validate <change-id> --strict`
- 對應 targeted tests
- `npm run typecheck`
- 必要時 `npm run build`
- 更新該 change 的 `tasks.md`
- 獨立 git commit

Phase 5 則改驗證：

- `openspec validate --strict`
- `openspec list`
- `git status --short`

# 六線 v2 執行計畫

本文件承接 `17_下一階段主線整合與優先級建議.md` 與 `18_五線內容深度執行計畫.md`，把下一階段六條 v2 主線拆成可開案、可驗證、可分階段提交的工作批次。

目前基線：

- `openspec list` 顯示沒有 active changes
- `client-interface / client-persistence / game-mechanics` 已吸收上一批完成狀態
- 角色、玩家、NPC、怪物仍維持文字 token；像素化只推進 terrain / background
- 每條線完成後必須單獨驗證、更新 tasks，並獨立提交

## 1. 執行總順序

Status: completed and committed as the first v2 implementation slice.

- encounter 已有 pending / choice / reward / resolved flow，但多數事件仍是單次內容

- 定義 chain id、chain step、world memory tag 與 repeat / cooldown 規則
- 補第一批可重複 / 可延伸的中後期事件鏈

- `soul.worldMemoryTags` 作為跨輪迴穩定世界記憶，migration 會補齊與清理 malformed payload。

## 2. Sub-agent 分工

本批次採用「主線總控 + 專線 explorer / worker」模式：

- Encounter agent: 事件鏈、world memory、resolved event、pending UI、persistence
- Sect/world agent: 三宗後段章節、map-local NPC、world route hooks
- Workshop agent: mastery milestone、specialization leaf、recipe sink、UI cue
- Reincarnation agent: soul progression、build planner、perk / heirloom 分化、migration
- Pixel terrain agent: terrain/background production polish、actor-token guard、mobile readability
- Spec gate agent: OpenSpec、migration、archive、validation checklist

實作階段每條 change 再建立對應 worker。worker 必須只負責自己的 change 範圍，不得回退其他 change 或未追蹤工作區內容。

## 3. Explorer 盤點結論

### Encounter

- 可復用核心是 `data/encounters.ts`、`encounterSlice`、`encounterActions`、`timeActions`、`PendingEncounterPanel`
- v2 最小 scope 是 chain prereq、pending context、run-scoped memory 與可選 soul-level world memory
- 若 memory 要跨輪迴保存，不能放在 current encounter state，必須放在 `soul` 或明確的跨世摘要

### Sect / World

- 現況已完成 `task_04 -> 120/121 -> 130` 的第一輪世界章節
- v2 最小 scope 是 `130+` 每宗再補一段章節模板、map-local NPC 與 once-per-run milestone encounter
- Quest 只負責 route gate，Encounter 負責事件 / reward 語意，Workshop 只吃材料與 source cue

### Workshop

- 現況已完成 recipe、specialization tree、mastery、material source cue 與 migration
- v2 最小 scope 不需要新 migration，除非新增 recipe provenance、craft queue、mastery history 或材料來源 persisted state
- 重點是 mastery milestone、specialization leaf、route-specific sink 與可測的副產物 / 品質 / 輸出 cue

### Reincarnation

- 現況已完成 Soul Save、Merit、Lifetime Stats、Reincarnation Hall、Perks、Heirloom planner 與主動坐化
- v2 最小 scope 應新增 build lane / planner version / selected build identity，否則很快會被舊 `rebirthConfig` shape 限制
- migration 風險集中在舊 planner 選擇的保留、clamp 與非法配置清理

### Pixel Map

- 現況已完成正式 terrain/background pixelization 與 landmark motifs
- v2 最小 scope 只補 terrain/background production vocabulary，不碰 actor token
- 必須補 terrain-only 白名單與 actor-token guard，確認 helper 不輸出 `spriteId / actorToken / tokenLabel / hudCue / combatOverlay`

### Spec / Migration Gate

- 目前沒有 active changes，base specs 已是 current truth
- 每條新 change 必須明確標示是否碰 persisted state
- schema 變更必須有 idempotent migration、hydrate sanitize 與 regression
- 完成後必須先回寫 base specs，再 archive；若使用 `--skip-specs` 必須記錄理由

## 4. 驗證與提交規則

每個 Phase 完成時至少要跑：

- `openspec validate <change-id> --strict`
- 對應 targeted tests
- `npm run typecheck`
- 必要時 `npm run build`
- `git diff --check`
- 更新該 change 的 `tasks.md`
- 獨立 git commit

所有 Phase 完成後再跑：

- `openspec validate --specs --strict`
- `openspec list`
- `npm run typecheck`
- 一組跨線 targeted regression
- `npm run build`
